# Forge Event Model

> Version 0.1
>
> This document defines how Forge understands Events.
>
> The Event Model is the foundation for connectors, identity resolution, interpretation, memory, relationships, attention, and intentional action.

---

# 1. Purpose

Forge begins with Events.

An Event is the first structured representation of something that happened in reality.

Events are not memories.

Events are not opinions.

Events are not relationships.

Events are immutable observations that may later be interpreted, questioned, remembered, and acted upon.

---

# 2. Definition

An Event is an immutable observation that something occurred at a specific point in time.

Every Event should answer:

- What happened?
- When did it happen?
- Where did it come from?
- Who or what may be involved?

Events preserve history.

Interpretations create meaning.

Memories preserve durable understanding.

---

# 3. Core Rule

Events are immutable.

Once created, an Event is never edited.

If new information arrives, Forge creates another Event.

History is append-only.

This allows Forge to preserve truth while allowing understanding to evolve.

---

# 4. Event Lifecycle

Reality

↓

Connector or Manual Entry

↓

Raw Input

↓

Normalized Event

↓

Identity Resolution

↓

Interpretation

↓

Question, Memory, Relationship, or Attention

---

# 5. Required Fields

Every Event must contain:

## id

A unique identifier.

## type

The kind of thing that occurred.

Examples:

- email.received
- reddit.comment.received
- order.placed
- phone.call.completed
- meeting.attended
- task.completed
- memory.created

## occurredAt

The time the event occurred in reality.

This is not always the same as the time Forge learned about it.

## source

Where the event came from.

Examples:

- gmail
- reddit
- facebook
- instagram
- woocommerce
- manual
- forge

## payload

The original structured data needed to understand the event.

Payload may differ by source.

Payload is evidence, not interpretation.

---

# 6. Recommended Fields

Future Events should also support:

## receivedAt

When Forge received or created the Event.

## subjects

The people, contacts, organizations, workspaces, products, or other entities that may be involved.

## provenance

Information describing how the Event was created.

Examples:

- imported by connector
- manually entered
- generated by Forge
- generated by AI
- synced from API

## visibility

Who is allowed to see the Event.

## workspaceIds

The Workspaces the Event may relate to.

## confidence

How confident Forge is that the Event is normalized correctly.

---

# 7. Events vs Payload

The Event is normalized.

The payload is source-specific.

Example:

An email from Gmail and a Reddit comment may have very different payloads, but both can become Events.

Forge should not allow UI, AI, or relationship logic to depend directly on raw connector payloads unless absolutely necessary.

Connector payloads are evidence.

Events are the normalized record.

---

# 8. Subjects

A Subject is anything the Event may be about.

Possible subjects:

- Person
- Contact
- Organization
- Workspace
- Product
- Task
- Relationship
- Unknown Identity

Subjects may be unresolved.

An Event may initially reference a Contact or Unknown Identity, then later become associated with a Person after identity resolution.

Events do not need perfect identity at creation time.

Forge learns over time.

---

# 9. Participants

Participants are people or entities that actively took part in the Event.

Examples:

In an email:

- sender
- recipients

In a phone call:

- caller
- recipient

In a meeting:

- attendees

In an order:

- buyer
- seller

Participants are a special type of Subject.

---

# 10. Event Types

Event types should be namespaced.

Recommended format:

```txt
domain.action.detail
```

Examples:

```txt
email.received
email.sent
reddit.comment.received
reddit.message.received
order.placed
order.refunded
task.created
task.completed
memory.created
question.answered
relationship.updated
```

Event types should describe what happened, not what it means.

Bad:

```txt
customer.interested
dealer.ready
relationship.strong
```

Good:

```txt
email.received
proposal.opened
order.placed
reply.sent
```

Meaning belongs to Interpretation.

---

# 11. Manual Events

Not all Events come from connectors.

Operators may manually create Events.

Examples:

- Had lunch with Doug.
- Called Mom.
- Installed cue tip for a player.
- Met dealer at trade show.
- Promised to send samples.

Manual Events are still Events.

They are immutable once created.

If the operator entered something incorrectly, Forge should create a correction Event rather than editing history.

---

# 12. Forge Events

Forge may create Events about its own behavior.

Examples:

- question.created
- question.answered
- interpretation.accepted
- memory.updated
- attention.dismissed
- task.completed

Forge Events are important because the operator's interaction with Forge becomes part of the history.

Forge should remember not only what happened in the world, but how the operator responded.

---

# 13. Correction Events

Because Events are immutable, corrections must be represented as new Events.

Example:

Original Event:

```txt
meeting.attended
```

Correction Event:

```txt
event.corrected
```

The correction may state:

- which Event was incorrect
- what was incorrect
- what the corrected understanding should be

The original Event remains.

History remains inspectable.

---

# 14. Event Confidence

Events may have confidence.

This does not mean the event did or did not happen.

It means Forge has some uncertainty about normalization.

Example:

Forge receives an email from:

```txt
john@example.com
```

The Event itself is real.

But Forge may be uncertain whether John is:

- John Smith
- John at CueStix
- an unknown person

That uncertainty belongs to identity resolution or interpretation, not the existence of the Event.

---

# 15. Events and Questions

When an Event cannot be confidently interpreted, Forge may create a Question.

Example:

Event:

```txt
email.received
```

Question:

```txt
Who is this contact?
```

Questions are not failures.

Questions are part of Forge's learning process.

Forge should ask the smallest useful question that improves future understanding.

---

# 16. Events and Interpretation

Events do not determine meaning by themselves.

An Event may support many Interpretations.

Example:

Event:

```txt
Doug said he switched back to Monster.
```

Possible Interpretation:

```txt
Doug currently prefers Monster.
```

Future Events may change that Interpretation.

The Event does not change.

---

# 17. Events and Memory

Events may support Memories.

A Memory is durable understanding retained by the operator.

Example Events:

- Observed drinking Monster.
- Observed drinking Red Bull.
- Discussed switching to Rockstar.
- Discussed returning to Monster.

Memory:

```txt
He has changed energy drink preferences several times. Current preference appears to be Monster.
```

Events preserve the moments.

Memory preserves the current understanding.

---

# 18. Events and Relationships

Relationships emerge from accumulated Events, Interpretations, Memories, and Intentional Actions.

No single Event defines a Relationship.

Events are evidence within a larger story.

Forge should avoid overreacting to isolated Events unless the Event has very high significance.

---

# 19. Events and Attention

Events may influence Attention.

However, an Event should not automatically interrupt the operator.

The Attention Engine decides whether an Event deserves attention based on:

- operator intention
- relationship importance
- time sensitivity
- confidence
- opportunity
- commitment
- context

Events create possibilities.

Attention determines what matters now.

---

# 20. Anti-Patterns

Forge should avoid the following:

## Treating unread as important

Unread does not mean meaningful.

## Treating recent as important

Recent does not mean valuable.

## Treating frequent as strong

Frequent communication does not always mean relationship health.

## Treating payload as domain

Raw connector data should not leak into core domain logic.

## Treating AI interpretation as fact

Interpretation is not Event.

## Editing history

Events are append-only.

---

# 21. Example

## Input

A Reddit connector receives:

```json
{
  "author": "poolguy92",
  "body": "Those blue tips look interesting. Are they harder than Kamui medium?",
  "createdUtc": 1782949321,
  "subreddit": "billiards"
}
```

## Normalized Event

```json
{
  "id": "evt_123",
  "type": "reddit.comment.received",
  "occurredAt": "2026-07-01T20:02:01.000Z",
  "source": "reddit",
  "subjects": [
    {
      "kind": "contact",
      "type": "reddit",
      "value": "poolguy92"
    },
    {
      "kind": "workspace",
      "id": "polydigm"
    }
  ],
  "payload": {
    "author": "poolguy92",
    "body": "Those blue tips look interesting. Are they harder than Kamui medium?",
    "subreddit": "billiards"
  }
}
```

## Possible Question

```txt
Who is Reddit user poolguy92?
```

## Possible Interpretation

```txt
This person is comparing Polydigm tips against Kamui and may be interested in hardness guidance.
```

## Possible Memory

```txt
This person has shown interest in Polydigm blue tips and Kamui comparisons.
```

## Possible Attention Item

```txt
Impact 78: Reply with hardness comparison and invite them to test a sample.
```

---

# 22. Implementation Guidance

The Event Model should be implemented in stages.

## Stage 1

Minimal Event:

- id
- type
- occurredAt
- source
- payload

## Stage 2

Add:

- receivedAt
- subjects
- provenance

## Stage 3

Add:

- visibility
- workspaceIds
- confidence

## Stage 4

Add:

- correction Events
- Event relationships
- audit trail

Do not overbuild Stage 4 before Stage 1 is useful.

---

# 23. Design Rule

When adding a new connector, ask:

1. What raw input does it produce?
2. What normalized Event should it create?
3. What subjects might be involved?
4. What identity questions might arise?
5. What interpretations might this Event support?
6. What memories might eventually be formed?
7. Could this Event deserve attention?

If those questions cannot be answered, the connector is not ready.

---

# 24. Summary

Events are the foundation of Forge.

They preserve what happened without pretending to know what it means.

Forge becomes intelligent not by collecting Events, but by learning how those Events shape Interpretation, Memory, Relationships, Attention, and Intentional Action.

Events are history.

Interpretations are understanding.

Memories are retained meaning.

Relationships are living stories.

Attention is finite.

Intentional Action is the point.
